# AI产品升级方案模板

## 技能用途

当用户要为某个**已有的AI产品/智能体**做功能扩展规划时，生成对外可分享的HTML方案。核心特点：
- 先问约束再开列：按产品实际可用的数据/系统分档，不强制零依赖
- 每个功能配真实业务场景举例，方便给决策层判断
- 自动脱敏（去公司名/品牌名/部门名），适合内部分享或外发

## 两阶段研发流程（v1.3 · 2026-07-17 新增）

**所有产品需求分两个阶段推进，不可跳过：**

### 第一阶段：概要设计（评审用）

| 维度 | 规范 |
|------|------|
| **目的** | 向领导/团队汇报，获取方向性确认和资源承诺 |
| **主文档** | HTML（人类阅读 + mermaid 旅程图 + 线框概念图） |
| **辅文档** | 纯 MD（AI 友好，供后续 AI 读取和续写） |
| **命名** | `产品名_概要设计_v0.1.html` + `产品名_概要设计_v0.1.md` |
| **模板** | `templates/概要设计_通用模板.html`（9章结构） |
| **深度** | 方案级：讲清楚做什么、为什么、大概怎么做 |
| **产出** | 9章文档（背景→功能→旅程→交互→技术→验收→问题→概念图→下一步） |

**此阶段文档统一叫「概要设计」。**

### 第二阶段：需求设计（开发用）

| 维度 | 规范 |
|------|------|
| **前置条件** | 概要设计已完成评审并封板 |
| **目的** | 给研发可执行的需求规格，每个功能可独立开工 |
| **主文档** | 纯 MD（详细设计，面向研发和 AI） |
| **辅文档** | HTML（需要时生成，如给领导看概要） |
| **命名** | `产品名_需求设计_v0.1.md`（版本号从 0.1 重新起） |
| **深度** | 规格级：每个字段、每个 API、每个边界条件 |

> ⚠️ **流程铁律**：概要设计未封板之前，**禁止进入需求设计阶段**。两个阶段之间明确的封版节点。

## 触发条件

- 用户说「XX助手/工具还能扩展什么功能」
- 用户说「给XX做升级方案」「XX下一步怎么做」
- 用户说「你看看竞品都在做什么」「竞品对标」
- 用户要跟领导/团队汇报产品规划
- 用户明确说「零依赖」「轻量做」或相反「我们有数据可以用」
- 用户说「概要设计」「需求设计」「封板」— 进入对应阶段

## 工作流

### 步骤1：明确产品现状和约束（必须问清楚再开始）
关键问题：
1. 产品当前的核心能力是什么？
2. **哪些企业数据/系统已经能接？**（LDAP？历史记录？知识库？OA/CRM？）
3. 用户希望走轻量路线还是深度集成路线？
4. 目标用户是公司内部还是外部？
5. 给研发看还是给决策层看？

> ⚠️ **不要默认零依赖**：邮件场景是用户主动选择轻量化，不代表所有产品都这样。先问清楚再分级。

### 步骤2：竞品对标
固定调研三类竞品：
- 国外标杆（Grammarly、Notion AI、Outlook Copilot 等）
- 国内同类型（网易邮箱AI、腾讯AI、WPS AI、钉钉智能助理 等）
- 跨行业借鉴（写作工具、办公套件）

每个竞品关注三个维度：
- 已经上线的功能
- 用户评价/付费转化点
- 哪些是零依赖就能做、哪些需要数据接入

### 步骤3：按约束分级（最重要）
**根据步骤1确认的数据可用性，功能分三档：**

| 档位 | 条件 | 典型功能 | 落地周期 |
|------|------|---------|---------|
| **轻量档** | 纯LLM+前端，0企业依赖 | 语气检测、风格切换、啰嗦精简、模板库 | 1-2周 |
| **数据档** | 可接企业数据（LDAP、知识库、历史记录） | 角色自适应、个性化记忆、智能跟进、知识增强 | 3-6周 |
| **集成档** | 可接外部系统（OA/CRM/ERP） | 审批联动、系统推送、跨平台打通 | 2个月+ |

同一个产品里三档可以并存——先出轻量档快速上线，再逐步加数据档和集成档。

### 步骤4：业务场景举例（决定方案成败）
**关键**：每个功能必须配 1-2 个具体业务场景，包括：
- 场景描述（谁、对谁、什么场景）
- 原文（Before）—— 用户当前会怎么写
- AI处理后（After）—— AI怎么改

### 步骤5：生成HTML方案（概要设计阶段）

> ⚠️ **用图铁律：图要让人一眼看懂。** 一张图只讲一件事，不超过5个节点/气泡，不加注释块。看的人15秒内理解就不用重画。聊天框效果图、Before/After对比、步骤圆点都是好图。mermaid/SVG/聊天框效果图都可以用，但**不能画成密密麻麻的流程图——那是文档不是演示**。

**固定布局方式**：

1. **Header**：标题 + 受众 + 约束标签 + 一句话说明
2. **第0章·方案总览**（金字塔塔尖）：
   - 功能快照表（4列：编号 | 功能 | 一句话价值 | 预期效果）
   - 核心效果对照表（3列：指标 | Before现状 | After目标）
   - 关键决策点表（4列：编号 | 决策 | 推荐方案 | 不拍板的后果）
   - 总览区**不用图**，用表格替代——总览是给人30秒扫完的
3. **N个功能卡片**（每个功能独立一张`.card`）：
   - 卡片标题 + 副标题说明
   - 1-2个场景举例（`.before`红字 vs `.after`绿字）
   - 实现方式：纯LLM / 需要接口 / 需要数据（标注难度标签`.tag`）
   - **可以用聊天框效果图**（画真实对话流）或 `.step .n` 步骤圆点表达旅程
4. **Footer**：下一步动作（3-5个具体项）

**用图三原则**：
- ✅ **行**：一张图讲一件事（如聊天框对话、Before/After对比、4步圆点）
- ❌ **不行**：密密麻麻10个节点8个分支的流程图，读者要停下来盯着研究
- ❌ **不行**：一页堆3张以上的大图，过犹不及

**绝对禁止**：
- 用"4层拆解"（意图/Prompt/输出/呈现）（到需求设计阶段再展开）
- 在第0章总览放图（用表格代替）

**场景举例是决定方案成败的关键**。每个功能必须配 `.example-box`：`.before` 和 `.after` 对比。

### 步骤6：脱敏检查（强制）
- 输出前 grep 检查敏感字眼：公司名、品牌名、部门名
- 命中后用泛化词替换：`公司`、`我司`、`内部团队`、`XX部门`
- 详见 references/脱敏规范.md

### 步骤7：交付（多通道）
根据用户需求选合适通道：
- 本地 HTML：用户自己看、截图用
- Get笔记：用户个人知识库存档
- 在线部署：直接发链接给领导/团队
- 飞书文档：方便团队评论协作

## 输出文档标准结构（v1.4 · 2026-07-17 卡片式概要设计）

> 方法论来源：邮件润色助手_内部评审版（0张图，纯CSS卡片，领导一直在看具体内容）。金字塔原理：塔尖先给结论和全貌 → 塔身展开细节。

### 概要设计阶段（当前阶段）

> ⚠️ 此阶段**可以用图**，但每张图只讲一件事。好的图：聊天框对话流、Before/After对比、步骤圆点。坏的图：密密麻麻的10节点流程图。不用4层拆解。


**固定结构**：

| 区块 | 内容 | 形式 |
|------|------|------|
| Header | 标题 + 受众 + 约束标签 | 居中 |
| 第0章·方案总览 | 功能快照表 + 效果对照表 + 决策点表 | 3张表，无图 |
| N张功能卡片 | 每张card：标题 + 副标题 + 1-2个场景(Before/After) + 实现方式 | `.card` + `.example-box` |
| Footer | 下一步动作 3-5 项 | 列表 |

**场景举例规范**（最核心的呈现方式）：
```html
<div class="example-box">
  <div class="label">SCENE · 具体场景名</div>
  <div class="before">原文...</div>
  <div class="after">AI处理后...</div>
</div>
```

### 需求设计阶段（概要封板后）

- 优先MD详细设计
- 可在详细章节使用 mermaid 辅助说明（非必需）
- HTML 根据需要生成

### 禁止清单
- 用密密麻麻的流程图（10+节点/8+分支/挤满文字）——读者要停下来研究 = 坏图
- 一页堆3张以上的大图，过犹不及
- 在第0章总览放图（用表格代替）
- 在第2章功能模块用"4层拆解"（需求设计阶段才展开）
- 把"4层拆解+交互改造+技术实现"塞到概要设计里（那是需求设计的内容）

## 版本管理规范

- 每轮评审后用 `git commit` 存档，commit message 含版本号 + 关键变更
- 文件名含版本号和阶段标识：`产品名_概要设计_v0.1.html` → `v0.2` → `v0.3`
- 同时保留 HTML（人类看）和 MD（AI看）双版本
- 新版本**在上一版基础上修改**，不覆盖、不重写——历史可追溯
- **封板后进入需求设计阶段**，版本号从 v0.1 重新起：`产品名_需求设计_v0.1.md`
- HTML 为**人类汇报用**（含样式、mermaid 图、概念图），MD 为 **AI 读取用**（无样式、纯内容）

## 关键经验

- **先问约束再分级**：邮件场景是用户主动选零依赖，但内部有LDAP/知识库的产品完全可以上数据档功能——先确认再开列
- **不要默认零依赖**：用户反馈 "我们内部有好多数据可用，不能限制只提供零依赖方案"
- **场景举例要具体到人**：用"催王总交6月报表"而非"日常邮件"
- **汇总表要带"难度"列和"依赖"列**：研发看难度，决策层看周期
- **文件命名用"内部评审版"**：用户复盘反馈"领导汇报"太生硬
- **三档可并存**：轻量档先上拿反馈，数据档和集成档并行推进
