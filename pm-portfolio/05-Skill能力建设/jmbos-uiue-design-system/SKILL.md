## 工作流（9步）

> **严格按顺序执行，禁止跳步/并步/绕过。** 所有样式值必须基于 `design-tokens.css`。

### 1. 输入解析

> **强制触发规则**：本技能不仅是首次生成时使用，**后续每次对已产出HTML的任何修改（样式调整、布局变更、组件替换、视觉优化、bug修复等）都必须重新触发本技能并以`--update`模式进入**。禁止绕过本技能直接修改代码文件。

**支持输入**：Figma URL / 需求名称 / 文件路径 / 粘贴文本 / `--update`增量模式

**路径构建**：
- 需求文档：`01.产品迭代工作区/{迭代}/1.产品/需求/{需求名称}.md`
- 正式输出：`01.产品迭代工作区/{迭代}/2.UI/设计交付/`
- 草稿：`01.产品迭代工作区/{迭代}/2.UI/草稿箱/`

**增量模式**：读取历史设计 → 变更分析 → **跳过步骤2-3** → 进入步骤4

**Figma URL 特殊处理**：MCP获取设计数据 → 下载图片到 `ASSETS/images/` → 解析组件/布局/样式 → 作为"需求文档"进入后续流程（须完成步骤6确认）

> **Figma 优先级原则**：Figma 设计稿（设计师输出）优先级 **高于** 规范样式描述。Figma 明确标注的严格还原；未明确细节用 design-tokens 兜底；资源层内联要求（Logo/Sprite/Tokens）属于技术约束，**规范强制**。执行逻辑：Figma 是"要做什么"（目标），规范是"怎么做"（方法）。

### 2-3. 历史变更判断与读取

- 涉及已有页面修改/调整/优化 → **进入步骤3**读取历史设计
- 新功能/新增页面 → **跳过步骤3**
- 无法判断 → 询问用户

### 4. 需求解析与分析

**4.1 提取**：页面类型、功能模块、业务规则、交互状态、异常边界

**4.2 质量**：可实现性判断、冗余识别、冲突识别

**4.3 疑点清单**：有关键缺失时向用户提问（无问题不产出）

**4.4 页面范围**：扫描所有独立页面，输出《页面范围清单》请用户确认

#### 4.5 多页面交互识别（≥2 页面时必执行，单页面跳过）

梳理页面间跳转/调用/数据传递关系，输出《交互矩阵》（含：触发页/动作/目标页/类型/参数传递）。类型：页面跳转 / 弹窗打开 / 抽屉打开 / 页内刷新 / 数据回填。≥3页面且交互≥5条时补充 Mermaid 流程图。

#### 4.6 规范预读取（未完成禁止进入步骤5）

**目标**：基于需求识别所需组件 → 逐个完整读取规范 → 输出读取证明

> **分段模式提醒**：本步骤为全局初始清单。**每页生成前必须重新执行页面级规范重读**（见步骤7.5），禁止仅依赖本步骤直接生成后续页面。

**流程**：识别所有组件（基础骨架+业务组件）→ 逐个阅读 `.md`/`.png` 提取3-5关键要点 → 输出《规范读取证明》（每项含：组件名/规范文件/关键要点，N/N完成后准予进入步骤5）

**防遗漏机制**：
1. **基础兜底**：即使认为"简单"，也须至少读取 components/README.md 对应页面类型的基础核心组件（列表≥7，详情≥5，表单≥6，弹窗≥6）
2. **全量扫描**：输出前浏览 ASSETS/README.md §三 的48组件清单
3. **人工确认**：步骤6请用户审核《组件完整清单》，可补充遗漏

#### 4.7 页面类型判定（逐项确认环节）

> **本步骤输出需在步骤6中由设计师逐一确认，未确认禁止进入步骤5。**

| 类型 | 定义 | 判定依据 | 后续流程 |
|------|------|---------|---------|
| **标准页面** | 完全匹配4种标准骨架，组件均在48个已覆盖范围内 | 骨架匹配 + 组件全覆盖 | 步骤7路径A |
| **半自定义页面** | 骨架匹配标准类型，但含未覆盖的自定义模块 | 骨架匹配 + 存在自定义模块 | 路径A（标准区）+路径C（自定义区） |
| **全自定义页面** | 无法匹配任何标准骨架 | 骨架不匹配 | 步骤7路径C |

**输出**：《页面类型判定表》（每行：页面名/类型/依据/设计师☐）。关键规则：多页面独立判定，允许混合类型；边界情况归入半自定义；判定结果直接影响步骤5和步骤7的路径选择。

### 5. 设计策略制定

> **判定粒度为「模块/区域级别」，非页面级别。策略选择取决于步骤4.7的页面类型判定结果。**

| 层级 | 范围 | 生成方式 | 规范依据 | 适用页面类型 |
|------|------|---------|---------|-------------|
| **页面骨架** | 列表页/详情页/表单页/弹窗(含抽屉) | **严格遵循标准骨架** | [page-list.md](REFERENCES/page-list.md) | 标准页面 / 半自定义页面 |
| **标准模块** | 48个已覆盖组件 | **严格基于对应组件规范** | [components/README.md](REFERENCES/components/README.md) | 标准页面 / 半自定义页面（仅标准区域） |
| **自定义模块** | 图表、统计卡等未覆盖区域 | **仅遵守Design Token** | [design-tokens.css](ASSETS/design-tokens.css) | 半自定义页面（自定义区域）/ 全自定义页面 |

**铁律（仅适用于标准页面及半自定义页面的标准区域）**：页面骨架和标准模块代码必须从规范源文件**逐行复制**，禁止凭记忆编写、浏览速查自行写CSS、用Token名替代固定值、融合改编多来源代码。

**自定义模块规则（适用于全自定义页面及半自定义页面的自定义区域）**：
- 颜色/字体/间距/圆角/阴影/过渡动画 必须用 CSS 变量（Design Token）
- 布局结构可灵活设计，不要求匹配标准骨架
- 组件实现可自行编写，不要求从规范源文件逐行复制
- 视觉风格需与 JMBOS 设计语言保持一致（通过 Token 保障）

**查阅顺序**：components/README.md → page-list.md → design-tokens.css → implementation-guide.md §0 → ASSETS/README.md
> 注：全自定义页面可跳过前两项，直接从 design-tokens.css 开始。

### 6. 方案确认（核心关卡 — 未通过绝对禁止进入步骤7）

> **反跳步检查点**：历史修改类需求必须先完成步骤3+4。

**确认覆盖项**：页面类型判定（4.7产出） / 整体布局方式 / 所有组件及变体 / 规范未覆盖的特殊处理 / 模糊矛盾缺失部分的取舍 / 未覆盖组件图标的生成方案

> **页面类型判定为逐项必确认项**：需将4.7产出的《页面类型判定表》提交设计师逐一确认（每行☐打勾），任一页面类型存疑则不进入步骤7。

**强制产出《组件完整清单》**：

| # | 组件名称 | 使用位置 | 变体/尺寸/状态 | 规范来源 | 状态 |
|---|---------|---------|---------------|---------|------|
| 1 | 导航菜单 | 页面左侧 | 侧边栏/180px宽 | navigation.md | ☐ |

> 涵盖每个使用的组件（含图标）；初始均为 ☐，步骤7.2逐一打勾；增量模式仅列变更项，未变更标注"沿用历史"

**主动确认**：设计稿模糊→列疑点 | 多种实现→说明优劣 | 规范未明确→提建议 | 需求有歧义→澄清

#### 6.1 多页面交互方案（≥2 页面时必填）

基于4.5交互矩阵，逐项确认实现方案，输出《交互串联方案》（分三类：跳转类含实现方式/目标文件/参数方式；弹窗/抽屉类含触发元素/数据回填；回调类含触发时机/目标行为；每项带☑）。单页面需求输出「无跨页面交互，跳过」即可。

### 7. 设计输出与实现

> **前置条件**：步骤6已通过。发现遗漏/矛盾 → **暂停→回退步骤6补确认**
>
> **反跳步防护**："样式调整/小修改"反馈禁止直接改代码，须先确认：①变更范围 ②是否影响已决策 ③是否重走步骤6。仅纯错别字/不涉及UI的文案调整可跳过。

#### 7.1 选择骨架 & 搭建基础环境

> **路径选择依据步骤4.7的页面类型判定结果，不同类型走不同子流程。**

**路径路由**：

| 页面类型 | 执行路径 | 说明 |
|---------|---------|------|
| 标准页面 | **路径A** | 复制标准模板，全量遵循规范 |
| 半自定义页面 | **路径A + 路径C（组合）** | 标准区域用路径A，自定义区域用路径C规则 |
| 全自定义页面 | **路径C** | 简化流程，仅遵守 Design Token 底线 |

##### 路径A — 标准页面 / 半自定义页面的标准区域

**步骤一：选择模板**

| 类型 | 模板文件 |
|------|---------|
| 列表页 | `page-shells/list.html` |
| 表单页 | `page-shells/form.html` |
| 详情页 | `page-shells/detail.html` |
| 弹窗 | `page-shells/modal.html` |
| 抽屉 | 复用 detail + `.drawer-overlay` + `.drawer-panel` |

**步骤二：搭建环境**

1. 所有资源内联，确保HTML独立运行无外部依赖
2. **Logo和图标Sprite是强制性资源，不得遗漏**
3. 禁止外联引用图标（`<use xlink:href>` / `<object>` / `<img>`）
4. Logo禁止硬编码色，必须用 `fill="var(--color-brand)"`

模板已预置三项完整资源（Design Tokens + 118个图标Sprite + JMBOS Logo）。使用前校验：`grep -o 'font-size-base: [0-9]*px' ASSETS/page-shells/list.html` 应返回13px，否则模板未同步禁止使用。

> **铁律（仅路径A生效）**：代码必须从规范源文件复制，禁止凭记忆编写、浏览速查自行写CSS、用Token名替代固定值。

---

##### 路径C — 全自定义页面 / 半自定义页面的自定义区域

**适用场景**：Dashboard大屏、可视化报表、特殊交互页、或任何标准骨架中嵌入的自定义模块区域。

**步骤一：新建 HTML 文件**

**步骤二：搭建基础环境（仅3项底线要求）**

| # | 操作 | 来源 | 强制性 |
|---|------|------|--------|
| C1 | 内联 Design Tokens 到 `<head><style>` | `ASSETS/design-tokens.css` | **强制** |
| C2 | 内联 SVG Sprite（`<svg style="display:none">` 放 `<body>` 开头） | `ASSETS/icons/icons-sprite.svg` | **推荐**（如需使用图标） |
| C3 | 内联 Logo SVG（`class="top-nav-logo"` + `fill="var(--color-brand)"`） | `ASSETS/brand/logo（支持替换颜色）.svg` | **按需**（页面含导航时必做） |

> **一键完成C1~C3**：`bash scripts/quick-fix.sh all <file.html>`
>
> **与路径A的区别**：路径C不要求从标准模板复制骨架，不要求逐行复制组件代码，不要求匹配4种标准布局。仅需保证 Design Token 内联以维持视觉一致性。

**步骤三：自动化验证**

```bash
bash scripts/validate-design.sh <文件.html>
```

| 检测项 | 标准 | FAIL修复 |
|--------|------|---------|
| E1 | Design Tokens 内联（`--font-size-base: 13px`） | `quick-fix.sh tokens <文件>` |
| E2 | SVG Sprite（使用了图标时需≥对应symbol数） | `quick-fix.sh sprite <文件>` |
| E3 | Logo（含导航时需内联SVG + 无硬编码色） | `quick-fix.sh logo <文件>` |

> **路径C验证宽松度**：E2/E3 按需检测——未使用图标可跳过E2，不含导航可跳过E3。但 E1（Design Tokens）为所有页面类型的硬性门槛。
>
> **一键修复+验证**：`bash scripts/quick-fix.sh all <文件.html> && bash scripts/validate-design.sh <文件.html>`

#### 7.5 输出控制

**原则**：单一职责 / 结构样式分离（禁内联样式除非动态计算）/ 异常覆盖 / 组件可复用

| 模式 | 条件 | 行为 |
|------|------|------|
| 单页面 | 1个页面 | 直接生成 |
| 分段 | ≥2个页面 | **每页独立执行完整规范读取流程** |
| 批量 | 用户要求全部 | 一次性生成 |

##### 分段模式（≥2页强制）

> **核心规则**：第2页起**必须重新执行规范读取**，禁止复用第一批上下文或记忆。

每页独立流程：A)输出《页面级规范读取证明-{页面名}》→ 空项则禁止继续 → B)执行7.1~7.4 → C)更新状态追踪表 → D)下一页回到A

**铁律（违反任一条=失败）**：禁止"上一页读过这页跳过"；禁止基于记忆编写代码；禁止证明为空或复制上一页；每页必须输出含具体要点的独立证明；必须维护状态追踪表。

编号规则：全新D-001起连续 / 增量保留原号新增追加 / 综合模式全量重编D-001~D-XXX

#### 7.6 多页面交互联动（≥2 页面且 6.1 已确认时执行）

> **前置条件**：所有页面 7.1~7.4 均已完成。

**实现原则**：JS内联到各HTML确保独立运行；渐进增强不影响静态展示；跳转优先URL参数，复杂数据用localStorage；缺后端接口用Mock数据。

**三种交互类型JS实现**：
- 页面跳转：`window.open('detail.html?id='+id)` / `window.close()` / `new URLSearchParams(location.search).get('id')`
- 弹窗/抽屉：`style.display='flex'`打开 / `.value=rowData.xxx`回填 / `style.display='none'`关闭
- 操作回调：`closeModal(id)+refreshList()+showSuccessMessage(msg)`

**交付验证**：输出《交互联动证明》（每项含：交互描述/源→目标/已验证☑），浏览器手动验证每个交互点后进入步骤8。

## 闭环与迭代规则

设计不完整不结束；自检未通过禁止输出至 `设计交付/`（草稿箱除外）；所有决策须有据可查。

> **每次修改必须重新触发本技能（`--update`模式）**，以下场景无一例外：
> - 样式微调（颜色/间距/字号/圆角等任何CSS变更）
> - 布局调整（增删模块、改变排列、响应式修改）
> - 组件替换或变体切换
> - 交互逻辑变更（弹窗/跳转/回调等JS改动）
> - 视觉优化或设计师反馈的任何调整
> - bug修复涉及UI呈现的
>
> **禁止行为**：收到上述任一请求时直接编辑HTML文件而不触发本技能。正确做法：触发技能 → `--update`模式 → 走完流程后输出。

反馈处理：设计师→直接执行；产品/开发/QA涉及视觉变更→先同步设计师确认；自行发现的规范偏差→记录并提交设计师确认。每次修改后重新执行步骤8。
