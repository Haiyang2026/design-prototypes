# 审批流程图绘制

绘制SVG格式的审批流程图，并嵌入到流程需求文档中。

触发场景：
- 新建流程需求文档时，需要绘制流程审批图
- 流程变更时，需要同步更新流程图（必须与文档同步更新）

---

## 一、准备工作

### 1.1 读取流程环节信息

先读取需求文档中的【审批步骤总览】表格，获取以下信息：

- 步骤ID和环节名称
- 环节属性（审批/执行/知会）
- 审批方式（单审/并审/串审/机器自动）
- 审批人获取方式（人员汇报关系/流程角色等）
- 退回配置
- 传阅/邮件配置

### 1.2 确定流程结构

按主流程/子流程区分：
- **主流程**：使用横向布局（从左往右），节点数一般不超过10个
- **子流程**：使用纵向布局（从上往下），如有并行分支用菱形网关

---

## 二、SVG绘制规范

### 2.1 画布尺寸

| 布局 | 画布宽 | 画布高 | 适用场景 |
|------|--------|--------|----------|
| 横向（主流程） | 1400 | 320 | 6-8个节点，无分支 |
| 纵向（子流程） | 1200 | 540 | 8-10个节点，含并行分支 |

超出范围时按比例扩展。

### 2.2 背景配色

| 元素 | 颜色 | 代码 |
|------|------|------|
| 画布背景 | 浅灰 | #f8f9fa |
| 图例背景 | 白色 | #ffffff |
| 图例边框 | 浅灰 | #e8e8e8 |

### 2.3 节点着色规范

| 节点类型 | 填充色 | 边框色 | 示例 |
|----------|--------|--------|------|
| 已审批/通过（单审） | #f6ffed | #b7eb8f | 发起人、直接主管 |
| 会签/待审批（并签） | #fff2f0 | #ffccc7 | 评审确认、工厂环节 |
| 系统自动/执行 | #f0f5ff | #adc6ff | 等待评审回复、机器自动发起 |
| 普通处理 | #fffbe6 | #ffe58f | 系统数据自动处理、数据回填 |

### 2.4 文字与字体

| 元素 | 字体 | 字号 | 颜色 |
|------|------|------|------|
| 主标题 | Microsoft YaHei, Bold | 17px | #1a1a1a |
| 副标题 | Microsoft YaHei | 11px | #999 |
| 节点名称 | Microsoft YaHei, Bold | 13px | #262626 |
| 节点备注 | Microsoft YaHei | 10px | #8c8c8c |
| 底部说明 | Microsoft YaHei | 11px | #999 |

### 2.5 节点尺寸

| 类型 | 尺寸 | 说明 |
|------|------|------|
| 审批节点矩形 | 宽120×高64（标准）/ 按内容扩展 | rx="8" 圆角 |
| 开始/结束圆环 | r="16"（外圈）r="7"（内圈） | 绿色实心=开始，红色空心=结束 |
| 并行网关菱形 | 宽40×高40（20×20） | 含"✕"标记 |
| 箭头 | 线宽1.5 | 灰色#999，三角箭头 |

### 2.6 审批助手标注

审批助手状态标注在对应节点下方：
- ✅ 审批助手（绿色#389e0d）→ 有审批助手
- ❌ 无审批助手（灰色#999）→ 无审批助手

## 三、并行网关绘制

适用于子流程有并行分支的场景。

### 3.1 结构

```
主流程 → 【分支网关◇】→ 上分支节点 → 【汇聚网关◇】→ 回填
                 ↘ 下分支节点 ↗
```

### 3.2 分支标签

分支上方标注文字说明分支含义，如"南安陶瓷评审线"、"永春良瓷评审线"。

### 3.3 节点数量

每条并行分支可包含多个节点（如对接人→领导），节点间用箭头连接。

## 四、文件规范

### 4.1 命名规则

```
[流程名称]_审批流程图_V[版本号].svg
```

示例：`智能品类二开工厂确认流程_审批流程图_V3.1.svg`

### 4.2 存储位置

SVG文件与流程需求文档放在**同一目录**下。

### 4.3 嵌入文档

在需求文档【审批步骤总览】后，添加如下Markdown引用：

```markdown
#### 2.1.1 审批流程图

![审批流程图说明文字](文件名.svg)
```

### 4.4 生成时机（重要）

**文档全部确认后才出流程图**，不在草案/演化阶段频繁生成。

| 场景 | 是否生成 |
|------|----------|
| 文档草案阶段、逐轮订正中 | **不生成**，等全文档确认后一次性出图 |
| 文档全部字段/流程/规则确认完毕 | **向用户确认**"现在可以生成流程图了吗？"，用户同意后再生成 |
| 后续需求变更导致流程环节变化 | 先确认变更内容，再同步更新流程图 |

> **原因**：避免因中间版本频繁改图，导致流程图版本号膨胀、样式漂移。

### 4.5 版本同步

流程图维护一个版本，与文档当前版本一致。不因文档小版本号升级而自动滚动。

### 4.6 样式约束（严格遵守）

- **风格模板**：以 `CAE分析申请流程_审批流程图_V1.1.svg` 为标准模板（浅色背景 #f8f9fa、绿=审批/蓝=执行、淡色填充+彩色边框、节点内三行文字、底部流程说明+图例、圆开始/圆结束）
- **禁止事项**：
  - 节点内禁止加入字段列表（如"└ CAE分析工程师"）
  - 禁止加入 "⚠ 必须填写"、"👥 多人并审" 等额外标签
  - 禁止改变节点尺寸来适应额外内容
  - 禁止添加非标准颜色/渐变填充

---

## 五、流程变动同步规则（重要）

当流程发生变动时，执行以下步骤：

1. **读取变更内容**：审核步骤总览中环节的增删改
2. **更新流程图**：重新生成SVG，同步反映所有变化：
   - 新增/删除节点
   - 节点顺序变化
   - 审批方式变化（单审→并审等）
   - 并行分支变化
   - 退回/回填规则变化
3. **更新版本号**：V[N] → V[N+1]
4. **更新变更文档**：在变更记录中说明流程图更新内容
5. **重新嵌入文档**：更新需求文档中的图片引用
