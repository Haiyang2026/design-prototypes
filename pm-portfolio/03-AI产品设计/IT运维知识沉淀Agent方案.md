## 一、Get笔记的 Agent 设计 —— 关键经验

Get笔记的 Skill 本质上做了这几件事，对运维场景有直接参考价值：

| 维度 | Get笔记的做法 | 对IT运维Agent的启示 |
|------|------------|-------------------|
| **意图路由** | 用一张路由表把自然语言（"记一下"/"搜一下"）映射到具体API | 做意图分类：重置密码 / 网络故障 / 软件安装 / 权限申请 |
| **上下文记忆** | 每次调用前检查配置/认证状态，有缺失就停下来问 | 维护工单状态、用户部门、历史问题记录 |
| **反幻觉边界** | 禁止编造ID、禁止跳过轮询、禁止伪造API响应 | 不能凭空说"已修复"，必须有工单系统的确认回执 |
| **失败重试** | 区分同步/异步任务，定义了明确的重试策略 | 运维操作（重启、开账号）同样要有超时和回滚机制 |

关键设计原则：**每次交互都强制产出结构化记录，不允许无记录的操作。**

## 四、如何落地：三阶段路线图

### 第一阶段（1-2个月）：搭骨架

- [ ] 接入企业微信机器人或钉钉机器人作为识别入口
- [ ] 建立IT运维知识库（腾讯文档 / Confluence / Notion），导入历史工单精华
- [ ] 梳理高频问题Top50，按 "问题 → 根因 → 解决步骤 → 验证方式" 四元组格式整理
- [ ] 用提示词工程（Prompt）做基础问答，评估覆盖率

### 第二阶段（2-4个月）：加肌肉

- [ ] 接入工单系统API，实现工单自动创建/关闭
- [ ] 对每次人工处理过的工单，自动提取"问答对"入库
- [ ] 建立"未命中题库"，运维人员每周补充答案
- [ ] 引入向量数据库（Milvus / Weaviate / ChromaDB），实现语义检索

### 第三阶段（持续）：提智能

- [ ] 统计未命中率下降曲线，用来说服管理层投入
- [ ] 对积累到1000+条高质量问答对后，考虑小模型微调
- [ ] 接入监控系统（Zabbix / Prometheus），实现主动告警 + 知识联动
- [ ] 完善Agent的安全边界：敏感操作二次确认、操作回滚机制

## 五、Prompt 设计要点

1. **反幻觉机制**：每次给出操作建议前，必须引用知识库来源
2. **工单绑定**：每个回答关联工单ID，确保可追溯
3. **转人工规则**：知识库无匹配时，不瞎猜，直接转人工并记录"未命中问题"
4. **操作指令生成**：User提问后，Agent生成可直接执行的运维命令（附带风险说明及回滚步骤）
5. **长期记忆**：针对高频用户，记住其部门/设备偏好，减少重复询问

## 七、关于图表生成的说明

### Auto 模式 vs DeepSeek

- **Auto 模式**：使用 WorkBuddy 内置的 Visualizer（SVG 矢量图）和 ImageGen（AI 位图）两种出图能力。SVG 图是代码渲染的，适合架构图/流程图，可无损缩放。
- **DeepSeek 模型**：可以通过 API 调用 ImageGen 工具生成图片，也可以直接让它输出 SVG 代码保存为 `.svg` 文件查看。但 DeepSeek 原生没有图像生成能力，需要依赖外部工具。
- **SVG 代码方式**（适用任何模型）：直接让模型生成 SVG 代码保存为文件，然后用浏览器打开即可看到图。本方案中的两张架构图就是用这种方式保存的。

### 三种出图方式总结

| 方式 | Auto 模式 | DeepSeek | 通用（任何模型） |
|------|-----------|----------|-----------------|
| SVG 矢量图 | 原生支持，直接渲染 | 写代码保存 `.svg` | 写代码保存 `.svg` |
| AI 位图(ImageGen) | 原生支持 | 调用工具 | 调用工具 |
| 腾讯文档Smart Page | 原生支持 | 通过MCP调用 | 通过MCP调用 |
| Mermaid 图 | 原生支持 | 输出代码 | 输出代码 |

> 结论：用 DeepSeek 也可以出图，只要让模型输出 SVG 代码保存成文件就行，效果一样。
