# 待办优先级规则文档

> 最后更新：2026-07-09
> 状态：草稿，仅包含已确认内容

---

## 1. 背景与问题

待办中心只有系统原始字段（标题、流程名称、当前环节、发起人、时间），缺少个人判断信息，无法通过标题判断事项的轻重缓急。

## 2. 已确认的优先级规则

### 规则一：反馈单最高优先级
- 用户反馈处理流程、iBPM 流程用户反馈 IT 流程 → 用户踢过来的，外部承诺压力大，优先干掉
- 不管有没有人催，反馈单天然排最高

### 规则二：催办信号
- **别人催你**：真的急，优先处理
- **你让申请人催办**：你在用催办功能变相实现「重点关注/星标」标记（因为待办中心没有这功能）

### 规则三：催办的 IT 业务需求单
- 需要先盯住，但处理周期长，不会很快关掉
- 属于"先关注再慢慢做"的类型

### 规则四：VIP 发起人天然优先
- 某些人的需求提上来就插队处理（如：张锦妙）
- 名单不固定，也会因为需求本身重要性而提升优先级

### 规则五：催办人权重
- 谁在催比有没有催更重要
- 催办人的身份/权重影响紧急程度

### 规则六：其余类型默认后排
- 工时评估单、权限申请等默认排在后面

### 规则七：自助类需求 → 默认「有空再做」
- 发起人 = 自己 + 无催办 + 无外部 deadline
- 有价值但不急，排进 TODO 但不能挤占反馈单和催办单的时间
- 示例：徐海洋自己提的新流程配置单

## 3. 个人优先级插槽设计

在系统字段之上，加一层个人专属覆盖层：

| 插槽字段 | 可选值 / 说明 |
|---------|------|
| 优先级标签 | 紧急 / 重要 / 普通 / 有空再做 |
| 关注标记 | ⭐ 重点关注（替代用催办 hack 的方式） |
| 个人备注 | 为什么重要、背景说明 |
| AI 建议 | 根据个人规则自动建议标签，人工确认 |

> 「有空再做」与「普通/低」的区别：普通是跟别的比靠后，「有空再做」是主动搁置、明确不急。

### 设计原则
- 个人插槽只对你自己可见，不影响其他人的待办视图
- 系统原始字段只读，插槽字段可读写

## 4. 团队扩展思路

此规则为小白（徐海洋）个人规则。给团队其他人使用时：

- 采用**苏格拉底式提问**，引导对方定义自己的优先级规则
- 每个人的规则独立存储，互不影响
- 核心框架（插槽结构、规则引擎）通用，规则内容个性化
